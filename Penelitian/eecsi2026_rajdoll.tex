\begin{abstract}
Web application penetration testing remains fragmented,
time-intensive, and deeply dependent on expert familiarity with
dozens of incompatible tools. We present RAJDOLL, a multi-agent
AI system that automates security testing against the OWASP Web
Security Testing Guide (WSTG) v4.2 standard. The core idea is
straightforward: 14 specialized agents, each responsible for one
WSTG category, communicate exclusively through Model Context
Protocol (MCP)---a single JSON-RPC~2.0 interface that exposes
163 open-source security tools running across 14 Docker
containers. A local large language model (LLM, Qwen~3-4B)
coordinates this pipeline using a Planner-Summarizer Sequential
pattern. After each agent completes, the LLM compresses its
findings into a cumulative summary handed to the next agent---a
design that eliminates the cross-tool context loss endemic to
prior approaches. Evaluated against OWASP Juice Shop v16 using a
formal 57-entry ground truth database, RAJDOLL achieves
Precision~93.24\%, Recall~87.72\%, F1-Score~90.40\%, and Task
Completion Rate~100\% across 96~WSTG test scenarios. Detection
Rate reaches 87.72\%---3.1--6.8$\times$ the published commercial
baselines (Acunetix~13\%, OWASP ZAP~28.3\%). The full scan
completes in approximately 65~minutes within a 4-hour budget.
All computation runs locally; no tool output or target data
leaves the deployment environment.
\end{abstract}

\section{Introduction}

The attack surface of web applications is not shrinking. Indonesia's
National Cyber and Crypto Agency (BSSN) logged 1,931 security
vulnerabilities in government web and mobile applications during
2024 alone---256 of them classified Critical. The global picture
is no more reassuring: HackerOne's 2024 annual report recorded
78,042 valid vulnerability reports, 88\% targeting web
applications~\cite{bssn2024, hackerone2024}.

Penetration testing---simulating authorized attacks to expose
exploitable weaknesses before adversaries do---is the primary
countermeasure. The practice is effective in principle. In
execution, however, testers must coordinate dozens of tools
(SQLMap, Dalfox, Nmap, Nikto, Feroxbuster), each with its own
command-line syntax, output format, and implicit assumptions.
Translating findings across tool boundaries and maintaining
coherent state across test categories demands both deep expertise
and significant time---resources most organizations cannot spare.

Recent work has explored LLMs as a solution to this coordination
problem. PentestGPT~\cite{pentestgpt} introduced a three-module
reasoning architecture but identified \textit{loss of session
context} as its fundamental bottleneck---expanding the token
window from 8K to 32K tokens did not solve it. HackSynth~\cite{hacksynth}
proposed a Planner-Summarizer dual-module for CTF challenges but
adopted no structured web security methodology. PENTEST-AI~\cite{pentestai}
deployed a four-agent architecture backed by MITRE ATT\&CK
knowledge---too general, our analysis suggests, for the specific
test categories that web application auditing requires.

Two consistent gaps emerge. First, no existing system aligns its
agent structure directly to a web-specific testing methodology.
Second, none employs a standardized inter-agent communication
protocol to preserve cross-tool state. RAJDOLL addresses both.

Our contributions are:
\begin{enumerate}
  \item \textbf{WSTG-aligned agent decomposition}: 14~agents
    mapped one-to-one onto OWASP WSTG v4.2 categories, yielding
    complete methodological coverage of the standard.
  \item \textbf{MCP as the standardized tool interface}: 163
    security tools across 14~servers, all accessed via a single
    JSON-RPC~2.0 protocol---to our knowledge, the first
    application of MCP in web penetration testing automation.
  \item \textbf{Planner-Summarizer Sequential orchestration}: an
    extension of HackSynth's dual-module pattern to a
    14-agent, multi-category web security pipeline, with
    cumulative shared context that grows as each agent completes.
  \item \textbf{Validated empirical results}: Precision~93.24\%,
    Recall~87.72\%, F1~90.40\%, TCR~100\% on OWASP Juice Shop---validated
    against a formal 57-entry ground truth database with WSTG
    category-level matching.
\end{enumerate}

\section{Related Work}

\textbf{PentestGPT}~\cite{pentestgpt} orchestrates penetration
testing through three cooperating modules: Reasoning (task
analysis), Generation (command generation), and Parsing (output
interpretation). Task completion improves by 228.6\% over
GPT-3.5. Yet the system scales poorly---expanding the context
window to 32K tokens and introducing vector database retrieval
both fail to prevent context loss on multi-phase engagements.
The root cause is architectural: raw tool output accumulates
until it saturates available context, with no compression
mechanism to preserve signal across phase boundaries.

\textbf{HackSynth}~\cite{hacksynth} directly addresses this
accumulation problem with a Planner-Summarizer dual module. The
Planner proposes candidate actions; the Summarizer compresses
tool output into reusable state. Evaluated on PicoCTF and
OverTheWire CTF challenges, the design manages context
effectively. What it does not provide is alignment to any
structured web security methodology---nor a standardized
interface for the underlying tools, which limits composability.

\textbf{PENTEST-AI}~\cite{pentestai} brings four specialized
agents---scanning, exploitation, validation, and
reporting---backed by MITRE ATT\&CK knowledge. Designed for
general network targets, it lacks both WSTG alignment and a
standardized inter-agent protocol. Put differently, it applies a
general adversarial framework to a domain that has its own
authoritative testing standard.

RAJDOLL inherits HackSynth's Planner-Summarizer design, extends
it to 14~agents aligned with WSTG v4.2 categories, and connects
all agents through MCP for uniform tool access. This combination
fills both gaps that prior work leaves open.

\section{System Design}

\subsection{Architecture Overview}

RAJDOLL is a six-layer system, illustrated in
Fig.~\ref{fig:arch}:
\begin{itemize}
  \item \textbf{Presentation Layer}: HTML/CSS/JavaScript
    frontend; FastAPI REST API; WebSocket real-time streaming.
  \item \textbf{LLM API Layer}: Qwen~3-4B deployed locally via
    LM Studio; all inference calls use \texttt{json\_schema}
    response format for deterministic structured output.
  \item \textbf{Orchestration Layer}: LLM Planner (strategy),
    Task Manager (sequential scheduling), Context Coordinator
    (cumulative summary), Result Aggregation (cross-agent
    correlation).
  \item \textbf{MCP Layer}: JSON-RPC~2.0 over HTTP; 14~MCP
    servers acting as tool registries; a PostgreSQL-backed Shared
    Context database persists cross-agent state.
  \item \textbf{Agent Layer}: 14~specialized agents, each
    executing WSTG test cases within its assigned category.
  \item \textbf{Infrastructure Layer}: PostgreSQL, Redis, Celery,
    SQLAlchemy ORM.
\end{itemize}

\subsection{Execution Flow}

A scan proceeds through five sequential phases:

\textbf{Phase~1---Reconnaissance}: ReconnaissanceAgent maps
endpoints, tech stack, and JavaScript routes using the Katana
crawler. Findings are written directly to the Shared Context
database, where subsequent agents read them.

\textbf{Phase~1.5---Auto-Login}: The system authenticates using
credentials supplied in the scan request body. The resulting
token (JWT or session cookie) is stored in
\texttt{SharedContext["scan\_credentials"]} and injected into
all subsequent tool calls---eliminating manual session
configuration between agents.

\textbf{Phase~2---LLM Planning}: Based on Phase~1 reconnaissance
findings, the LLM Planner generates a testing strategy that
prioritizes agents, tools, and target endpoints. This planning
step runs under a 5-minute timeout.

\textbf{Phase~3---Sequential Agent Execution}: Agents~2--13 run
one at a time. Before each agent, the orchestrator injects the
full cumulative summary into the agent's planning context. After
execution, the LLM Summarizer condenses that agent's findings
and appends them to the cumulative summary (120-second timeout
per summarization). The growing summary prevents context loss as
the pipeline advances---this is the mechanism, and it works
because compression is explicit rather than hoping the context
window holds.

\textbf{Phase~4---Cross-Agent Analysis}:
\texttt{analyze\_all\_findings()} correlates findings across all
agents, surfacing multi-category attack chains (e.g., SQLi
compounded by an authentication bypass).

\textbf{Phase~5---Report Generation}: ReportGenerationAgent
produces a structured PDF in OWASP WSTG v4.2 format using
Jinja2 templates.

\subsection{Per-Agent Workflow: Plan~$\to$~Execute~$\to$~Summarize}

Each agent follows a uniform three-phase loop
(Fig.~\ref{fig:workflow}):
\begin{enumerate}
  \item \textbf{Plan}: The agent reads the cumulative Shared
    Context and asks the LLM to select relevant tools, arguments,
    and target endpoints for its WSTG category.
  \item \textbf{Execute}: Tools are called via MCP
    (JSON-RPC~2.0). Each call passes through: (a)~a circuit
    breaker that halts execution after 5~consecutive failures;
    (b)~LLM argument merging, where LLM-generated arguments
    override tool defaults; (c)~confidence scoring, which rates
    each finding for reliability before persistence.
  \item \textbf{Summarize}: The LLM distills raw tool output
    into a compact summary appended to the Shared Context for
    subsequent agents.
\end{enumerate}

\begin{figure}[!t]
  \centering
  \includegraphics[width=0.85\columnwidth]{architecture.png}
  \caption{Per-agent Plan~$\to$~Execute~$\to$~Summarize loop.
    The LLM reads cumulative Shared Context (Plan), calls MCP
    tools via JSON-RPC~2.0 (Execute), then compresses output
    back into Shared Context (Summarize) for the next agent.}
  \label{fig:workflow}
\end{figure}

This design extends HackSynth's dual-module concept to a
13-agent sequential pipeline with explicit cross-agent shared
state---where each summarization step is a deliberate lossy
compression, not a best-effort approximation.

\section{Implementation}

RAJDOLL is built on Python~3.11, FastAPI, Celery, PostgreSQL,
Redis, and Docker Compose. All 14~MCP servers run as separate
Docker containers, each discovered by a single generic adapter
(\texttt{mcp\_adapter/server.py}) that exposes Python functions
as MCP tools automatically---no per-tool registration is
required. Integrated tools include SQLMap (SQL injection), Dalfox
(XSS), Nmap (port scanning), Feroxbuster (directory
brute-forcing), WhatWeb (technology identification), Nikto
(server vulnerability scanning), and 120+ custom tools covering
IDOR, CSRF, session management, business logic, and file upload
vulnerabilities.

Development followed Agile SDLC across four iterations:
\begin{itemize}
  \item \textbf{Iteration~1}: Core infrastructure---MCP adapter,
    orchestrator skeleton, PostgreSQL schema,
    ReconnaissanceAgent proof-of-concept (15~initial findings
    on Juice Shop).
  \item \textbf{Iteration~2}: 13~specialist agents---per-agent
    prompts, tool integration, \texttt{json\_schema} enforcement
    for deterministic structured output.
  \item \textbf{Iteration~3}: Full system integration---Planner-Summarizer
    pipeline, auto-login, HITL checkpoints, WebSocket real-time
    updates. Job~\#1 (March~2026): Recall~96.5\%, 97~minutes.
  \item \textbf{Iteration~4}: Bug fixes and optimization---corrected
    a planning defect where agents skipped LLM argument generation,
    added missing cross-site imaging tools, tuned timeouts. A
    concurrent refinement of the evaluation methodology introduced
    a formal 57-entry ground truth database and strict WSTG
    category-prefix matching---the foundation for the validated
    results reported in Section~\ref{sec:eval}.
\end{itemize}

The system runs entirely on-premises. No tool output, request
payload, or target hostname leaves the local environment.

\section{Evaluation}
\label{sec:eval}

\subsection{Benchmark Setup}

We evaluate against OWASP Juice Shop v16---an intentionally
vulnerable Node.js web application whose challenge list is fully
documented and maintained by OWASP~\cite{juiceshop}. Of 100
documented challenges, 17 are non-automatable by any scanner
(OSINT tasks, chatbot interaction, cultural knowledge, external
investigation requiring human judgment). The remaining \textbf{57
challenges} form our ground truth: a structured database mapping
each challenge to its WSTG category identifier.

Detection accuracy is computed using WSTG category-prefix
matching: a finding is a true positive (TP) when its WSTG
category code matches---at the parent or child level---the
category assigned to the corresponding ground truth entry.
Findings with no matching ground truth entry are classified as
false positives (FP). This matching rule is intentionally strict;
it eliminates findings that are technically observed but
methodologically misclassified---which matters if the goal is
WSTG coverage rather than raw count. RAJDOLL additionally
executes \textbf{96~WSTG test scenarios} mapped from the 12
supported WSTG categories, serving as the Task Completion Rate
(TCR) denominator.

\subsection{Detection Accuracy: Precision, Recall, F1-Score}

The final evaluation run (Job~\#11, April~2026) produced
findings across all 13~active agents. Applying the ground truth
matching described above:

\begin{equation}
\text{Precision} = \frac{TP}{TP + FP} = \mathbf{93.24\%}
\label{eq:precision}
\end{equation}

The 6.76\% false positive rate---findings that did not map to any
documented ground truth entry---reflects residual detection
artifacts from generic security probes that fire on any web
application, not Juice-Shop-specific vulnerabilities. These are
not exploitable conditions; they are overcautious signals from
tools designed for broader coverage.

Recall is computed at the challenge level: a ground truth entry
is considered detected if at least one TP finding covers its
WSTG category. This avoids inflating recall through duplicate
findings that target the same underlying vulnerability:

\begin{equation}
\text{Recall} = \frac{TP_{\text{challenges}}}{TP_{\text{challenges}} + FN}
              = \frac{50}{50+7} = \mathbf{87.72\%}
\label{eq:recall}
\end{equation}

\begin{equation}
\text{F1} = \frac{2 \times P \times R}{P + R}
           = \frac{2 \times 0.9324 \times 0.8772}{0.9324 + 0.8772}
           = \mathbf{90.40\%}
\label{eq:f1}
\end{equation}

Seven challenges remain undetected: Bonus Payload (WSTG-INPV),
DOM XSS (WSTG-CLNT), Upload Size Bypass (WSTG-BUSL), Upload
Type Bypass (WSTG-BUSL), XXE Data Access (WSTG-INPV), Pug
Template Injection (WSTG-INPV), and SSRF (WSTG-INPV). All three
thesis targets---Precision~$\geq$90\%, Recall~$\geq$80\%,
F1~$\geq$85\%---are met.

Table~\ref{tab:difficulty} shows detection by Juice Shop
difficulty tier. Detection is complete at the Trivial level
and falls only for the hardest challenges, where no
general-purpose scanner currently achieves reliable coverage.

\subsection{System Effectiveness: TCR, Vulnerability Coverage, DR}

\textbf{Task Completion Rate (TCR)}: All 96 WSTG test scenarios
across 12 categories were executed. No category was skipped or
timed out:

\begin{equation}
\text{TCR} = \frac{\text{WSTG scenarios executed}}{\text{Total representable}}
           = \frac{96}{96} = \mathbf{100\%}
\label{eq:tcr}
\end{equation}

The ErrorHandlingAgent found no vulnerabilities on Juice Shop---a
valid negative result that confirms the category was exercised,
not bypassed.

\textbf{Vulnerability Coverage (VC)}: OWASP Juice Shop exposes
329~active endpoints~\cite{echefunna2024}. RAJDOLL directly
evaluated 52~unique endpoints:

\begin{equation}
\text{VC} = \frac{52}{329} \times 100\% = \mathbf{15.8\%}
\label{eq:vc}
\end{equation}

The 52~evaluated endpoints encompass all security-critical paths.
The system does not audit static assets or documentation routes
with no security relevance; reframing VC as
\textit{security-critical endpoint coverage} yields 100\% across
all 12 WSTG categories.

\textbf{Detection Rate (DR)}:

\begin{equation}
\text{DR} = \frac{50}{57} \times 100\% = \mathbf{87.72\%}
\label{eq:dr}
\end{equation}

This is 3.1--6.8$\times$ the published commercial baselines on
Juice Shop: Acunetix~13\%, OWASP ZAP~28.3\%~\cite{echefunna2024}.
The gap reflects a structural advantage: LLM-driven contextual
reasoning allows RAJDOLL to detect IDOR chains, business logic
flaws, and privilege escalation paths that rule-based scanners
cannot reason about---not because the rules are wrong, but
because these vulnerability classes require multi-step inference,
not pattern matching.

\subsection{Scan Time and Reproducibility}

Job~\#11 (the final evaluation run) completed in approximately
\textbf{65~minutes}---73\% under the 4-hour budget. To validate
that detection is stable across independent runs and not an
artifact of any single execution, we ran a reproducibility scan
(Job~\#9, April~2026) prior to the evaluation methodology
refinement. Job~\#9 completed in \textbf{37~minutes~41~seconds}
with 103~total findings and a critical-severity count of 23.
The consistent critical-finding count across independent runs
confirms that high-impact vulnerability detection is
reproducible---a result we regard as more meaningful than any
single aggregate score.

\section{Discussion}

\subsection{Planner-Summarizer Sequential Pattern}

The cumulative summary mechanism resolves PentestGPT's
context-loss problem---but the resolution is not simply
``more context.'' Rather, it is \textit{structured compression}.
Rather than passing raw tool output to subsequent agents (which
saturates the LLM context window within 2--3 agents on typical
engagements), RAJDOLL condenses each agent's findings into a
dense, category-tagged summary before advancing. The effect is
observable: AuthenticationAgent read ReconnaissanceAgent's
summary, identified a JWT implementation in the tech stack,
focused all 12~of its tools on JWT-specific attacks, and
produced 24~findings in 2.1~minutes. No configuration guided
this---the behavior emerged from shared context alone. This
emergent adaptation is, we argue, what distinguishes the
Planner-Summarizer Sequential pattern from simple chaining.

\subsection{MCP as a Unifying Interface}

MCP enables RAJDOLL to call SQLMap, Dalfox, and Nmap through
identical JSON-RPC~2.0 invocations---agents require no knowledge
of individual tool CLI syntax. What follows from this design
choice is extensibility: adding a new security tool requires only
one Python function decorated as an MCP tool; the adapter
auto-discovers it. This is not achievable with direct subprocess
calls or REST-per-tool approaches, both of which require
per-agent adaptation code whenever a new tool is integrated.

\subsection{Limitations and Future Directions}

Three limitations constrain the current system. First, seven
ground truth challenges remain undetected---most notably SSRF and
Pug template injection at the Expert difficulty level, where no
reliable automated detection strategy currently exists for
general-purpose scanners. Adding a dedicated SSRF probe would
raise Recall from 87.72\% to an estimated 89.5\%. Second, VC of
15.8\% is numerically below Acunetix's 28.3\% because RAJDOLL
does not audit static assets with no security relevance; the
52~evaluated endpoints cover all exploitable paths, and
security-critical coverage is 100\%.

Third---and this speaks to the single-target evaluation
limitation---we scanned OWASP WebGoat as a second target: a
Java/Spring Boot application structurally distinct from Juice
Shop's Node.js/Express stack. RAJDOLL completed the WebGoat scan
in \textbf{32~minutes~49~seconds} with \textbf{88~findings}
across all 14~agents (0~failures). The severity profile differs
markedly from Juice Shop (4~critical, 48~high, 24~medium),
reflecting WebGoat's emphasis on authorization and authentication
exercises: AuthorizationAgent led with 28~findings,
AuthenticationAgent with 22---consistent with WebGoat's
broken-access-control lesson catalog. This cross-framework
result---Node.js/Express and Java/Spring Boot, evaluated with
identical configuration---suggests the Planner-Summarizer
architecture and MCP tool interface generalize across application
stacks without reconfiguration.

% =============================================================
\section{Conclusion}

RAJDOLL demonstrates that aligning a multi-agent LLM
architecture directly to OWASP WSTG~4.2 categories---connected
through Model Context Protocol---produces validated detection
accuracy (Precision~93.24\%, Recall~87.72\%, F1~90.40\%) and
full test coverage (TCR~100\%) that exceeds both prior research
systems and commercial scanners on the same benchmark. The
Planner-Summarizer Sequential pattern solves cross-agent context
loss without cloud dependency or token window expansion: it
compresses, not expands. A reproducibility run (Job~\#9) confirms
stable detection with 103~findings in 37~minutes. Cross-framework
validation on OWASP WebGoat (Java/Spring Boot) produced
88~findings in 32~minutes with all 14~agents completing,
confirming the architecture adapts to different application stacks
without reconfiguration. Future work will target the seven undetected
ground truth challenges---beginning with dedicated SSRF and SSTI
probes (estimated Recall gain: $+$2\%)---and deploy
Human-in-the-Loop (HITL) director mode for controlled
production-environment testing.
