# Design Spec: Report Enrichment & PDF Overhaul

**Date:** 2026-03-30
**Author:** Martua Raja Doli Pangaribuan
**Status:** Approved — ready for implementation

---

## Problem

The current RAJDOLL PDF report (`RAJDOLL_Report_Job1.pdf`) lists findings with only raw tool output. There are no:
- Explanations of what each vulnerability means
- Structured remediation steps
- References (OWASP WSTG, CWE, NIST)
- CVSS 4.0 scores
- Professional layout

Root cause: `add_finding()` stores raw evidence as `details`. The `remediation` field is hardcoded `None` in `_collect_findings()`. The PDF generator (`pdf_report.py`) is a 280-line f-string with no structure.

---

## Solution Overview

**Hybrid enrichment** at finding write time + **Jinja2 HTML template** rendered by WeasyPrint.

- 90% of findings: matched against a static JSON knowledge base (instant, deterministic)
- 10% of findings: LLM fallback via Jinja2 prompt template + `json_schema` enforcement
- PDF: `report.html.j2` Jinja2 template replaces inline f-string HTML generation
- Executive summary: uses `SharedContext["final_analysis"]` (already generated by orchestrator Phase 4)

---

## Architecture & Data Flow

```
add_finding(category, title, severity, evidence, details)
  → Finding saved to DB (raw columns unchanged)
  → EnrichmentService.enrich(category, title, severity, evidence)
      ├─ StaticKBMatcher.match(category, title)
      │    ├─ HIT  → EnrichmentResult (source="static_kb", instant)
      │    └─ MISS → LLMEnricher.enrich(category, title, severity, evidence)
      │               ├─ render enrichment_prompt.j2 with Jinja2
      │               ├─ call SimpleLLMClient (json_schema enforced)
      │               └─ EnrichmentResult (source="llm")
      └─ Finding.update(explanation, remediation, cwe_id, wstg_id, cvss_score_v4, references, enrichment_source)

GET /scans/{id}/report/pdf
  → load all Findings (enrichment columns populated)
  → load SharedContext["final_analysis"]
  → render report.html.j2 via Jinja2
  → WeasyPrint → PDF bytes → HTTP response
```

## Component Design

### EnrichmentResult (dataclass)

```python
@dataclass
class EnrichmentResult:
    explanation: str          # What this vulnerability is and why it matters (2-3 sentences)
    remediation: str          # Step-by-step fix as markdown numbered list
    cwe_id: str               # e.g. "CWE-89" (empty string if unknown)
    wstg_id: str              # e.g. "WSTG-INPV-05" (empty string if unknown)
    cvss_score_v4: float      # CVSS 4.0 base score (0.0–10.0), 0.0 if unknown
    references: list[str]     # ["https://owasp.org/...", "https://cwe.mitre.org/..."]
    source: str               # "static_kb" | "llm" | "fallback"
```

### EnrichmentService

- `enrich(category, title, severity, evidence) → EnrichmentResult`
- Calls `StaticKBMatcher.match()` first
- On miss: calls `LLMEnricher.enrich()`
- On any failure (timeout, JSON parse error, LLM unavailable): returns `_safe_fallback()` — never raises, never blocks a finding write
- `_safe_fallback()` returns generic text based on severity + empty refs, `source="fallback"`

### StaticKBMatcher

- Loads `enrichment_kb.json` once at module import (cached)
- Match logic: `category_pattern` substring match on `category` **AND** any `title_keywords` item found in `title.lower()`
- First match wins
- Returns `EnrichmentResult` with `source="static_kb"`

### Static KB Entry Schema (`enrichment_kb.json`)

```json
{
  "id": "sqli-generic",
  "category_pattern": "Input Validation",
  "title_keywords": ["sql injection", "sqli", "sql error", "blind sql"],
  "wstg_id": "WSTG-INPV-05",
  "cwe_id": "CWE-89",
  "cvss_score_v4": 9.3,
  "explanation": "SQL Injection allows attackers to manipulate backend database queries by injecting malicious SQL syntax through user-supplied inputs. Successful exploitation can lead to unauthorized data access, authentication bypass, data destruction, or full database server compromise.",
  "remediation": "1. Use parameterized queries or prepared statements for all database interactions.\n2. Apply strict input validation and allowlist filtering.\n3. Enforce least-privilege database accounts (no DROP/ALTER in app accounts).\n4. Enable WAF rules for SQL injection patterns.\n5. Audit and log all database query errors.",
  "references": [
    "https://owasp.org/www-project-web-security-testing-guide/v42/4-Web_Application_Security_Testing/07-Input_Validation_Testing/05-Testing_for_SQL_Injection",
    "https://cwe.mitre.org/data/definitions/89.html",
    "https://cheatsheetseries.owasp.org/cheatsheets/SQL_Injection_Prevention_Cheat_Sheet.html"
  ]
}
```

| WSTG Category | Entries |
|---------------|---------|
| WSTG-INPV | SQLi, XSS (reflected/stored/DOM), SSTI, LFI, NoSQL injection, HPP, ReDoS, CSRF |
| WSTG-ATHN | Brute force, weak password policy, 2FA bypass, default credentials |
| WSTG-ATHZ | IDOR, privilege escalation, BOLA, user spoofing |
| WSTG-SESS | Session fixation, token predictability, missing HttpOnly/Secure flags |
| WSTG-CONF | Missing security headers, directory listing, exposed .git, npm CVEs, hidden endpoints |
| WSTG-CLNT | Clickjacking, open redirect, CSP missing/weak, CORS misconfiguration |
| WSTG-BUSL | Rate limiting absent, coupon forgery, negative quantity abuse |
| WSTG-IDMG | User enumeration, mass assignment, insecure registration |
| WSTG-CRYP | Weak TLS, weak hashing (MD5/SHA1) |
| WSTG-ERRH | Stack trace exposure, verbose error messages |

### LLM Enrichment Prompt (`enrichment_prompt.j2`)

```jinja2
You are a security expert writing a professional penetration test report.
Analyze this vulnerability finding and return structured enrichment data.

Finding details:
- Category: {{ category }}
- Title: {{ title }}
- Severity: {{ severity }}
- Evidence summary: {{ evidence_summary }}

Return a JSON object with exactly these fields:
- "explanation": What this vulnerability is and its business impact (2-3 sentences, professional tone)
- "remediation": Step-by-step remediation as a numbered markdown list (minimum 3 steps)
- "cwe_id": Most applicable CWE ID string (e.g. "CWE-79"), or "" if unknown
- "wstg_id": Most applicable OWASP WSTG 4.2 test ID (e.g. "WSTG-CLNT-02"), or "" if unknown
- "cvss_score_v4": Estimated CVSS 4.0 base score as a float between 0.0 and 10.0
- "references": List of 2-3 relevant URLs (prefer OWASP WSTG, CWE, OWASP Cheat Sheet)
```

Enforced via `json_schema` — same pattern as `SimpleLLMClient.select_tools_for_agent()`.

```python
wstg_id           = Column(String(30),  nullable=True)   # "WSTG-INPV-05"
cvss_score_v4     = Column(Float,       nullable=True)   # CVSS 4.0 base score
references        = Column(JSON,        nullable=True)   # ["https://...", ...]
enrichment_source = Column(String(20),  nullable=True)   # "static_kb"|"llm"|"fallback"
```

Migration: new Alembic version file, `ADD COLUMN` only (no existing column changes, no data migration needed — existing findings get `NULL` enrichment columns, populated on next scan).

`UniqueConstraint("job_id", "agent_name", "category", "title")` unchanged.

## Error Handling

| Scenario | Behavior |
|----------|---------|
| KB JSON missing/corrupt | Log warning at import, all findings go to LLM path |
| LLM timeout | Return `_safe_fallback()`, `source="fallback"`, log warning |
| LLM returns invalid JSON | Retry once with stricter prompt, then `_safe_fallback()` |
| LLM unavailable (port 1234 down) | `_safe_fallback()` immediately, no retry |
| WeasyPrint missing | HTTP 500 with actionable install message |
| `final_analysis` missing from SharedContext | Executive summary renders without LLM narrative, no crash |
